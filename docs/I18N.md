# Internationalisation

The site ships in two locales, English and Arabic. The translation files are at the repository root: `messages/en.json` and `messages/ar.json`. Both are hand-authored. The Arabic version was written by a native speaker, not produced by a machine.

## What is in the messages files

Only UI chrome. Page titles, button labels, navigation, breadcrumb labels, category labels, and the four short paragraphs that describe the about page. The keys are organised into namespaces (`site`, `nav`, `home`, `timeline`, `era`, `event`, `breadcrumb`, `about`, `footer`, `notFound`, `categories`).

What is not in those files: event titles, event summaries, significance paragraphs, Quran verse text, Quran translations, hadith narrations, location names, key figures, primary source names. Those live in `data/events/*.json` and they are read directly. Event titles render in Arabic from `titleArabic` on the Arabic site and in English from `title` on the English site. Long-form fields like summaries and significance are presented in English on both locales because we have not added Arabic versions yet, and we will not add them by machine. If a future scholarly translation is authored, it will go into `data/events/*.json` as new fields.

## Adding a new UI string

1. Pick the right namespace, or add a new one if no existing namespace fits.
2. Add the key to `messages/en.json` with the English string.
3. Add the same key to `messages/ar.json` with a hand-authored Arabic translation. Do not paste machine output. If the project has no human translator available for a string, leave the key out and code around it.
4. In the consuming component, call `useTranslations('namespace')` and read with `t('key')`.

## The religious content rule

The application never produces, modifies, or retranslates Quran, hadith, or seerah text. This applies to placeholders, examples, fallbacks, and tests. If a key in `messages/*.json` would require restating religious content to translate, the right answer is usually to display the source field directly from `data/events/`, with `dir="rtl" lang="ar"` for Arabic spans, instead of putting the content into `messages/`.

## Routing

`next-intl` handles routing in `as-needed` mode. The default locale (English) is unprefixed; Arabic lives under `/ar`. So:

- English home: `/`
- Arabic home: `/ar`
- English event: `/era/meccan/mec-007`
- Arabic event: `/ar/era/meccan/mec-007`

Configuration is in `src/i18n/routing.ts`. The middleware at `src/proxy.ts` is a thin wrapper around `createMiddleware(routing)`. Internal pages should import `Link`, `useRouter`, and `usePathname` from `@/i18n/navigation` rather than directly from `next/navigation`. The next-intl wrappers preserve the current locale across navigation.

## Locale switcher

The header carries a button that toggles between locales. It reads the current pathname (without the locale prefix) and replaces the locale via `useRouter().replace(pathname, { locale: target })`. The path is preserved across the switch, so a user reading an event in Arabic at `/ar/era/meccan/mec-007` who taps "English" lands at `/era/meccan/mec-007` in English without losing their place.

## Dates and numerals

Date formatting is locale-aware. On English, dates render as the source string (e.g., "Ramadan, 12 BH (610 CE)"). On Arabic, the formatter at `src/lib/dates.ts` produces a Hijri-first form with Arabic month names, Eastern Arabic numerals, and the abbreviations `هـ` (after-Hijra) or `ق.هـ` (before-Hijra), with the Gregorian year in parentheses with `م`. The numerals helper `toEasternDigits` performs a mechanical substitution and is reused for badge dates.

## Direction and fonts

The locale layout sets `dir="rtl"` and `lang="ar"` on the `<html>` element for Arabic, and `dir="ltr" lang="en"` for English. The body font on Arabic is IBM Plex Sans Arabic; the body font on English is Source Serif 4. Amiri remains the Arabic family for Quranic verses and classical Arabic in the JSON, applied via the `font-arabic` utility class regardless of locale.

When mixing scripts in a single line (Arabic title next to a CE date, or a transliterated name in an Arabic paragraph), wrap the inner-direction content in a `<span dir="rtl" lang="ar">` or vice versa so bidi resolves cleanly. Logical CSS properties (`start`, `end`, `me`, `ms`, `pe`, `ps`) handle the rest. The timeline spine, breadcrumb chevrons, and prev/next arrows all flip on RTL automatically through `rtl:scale-x-[-1]`.
